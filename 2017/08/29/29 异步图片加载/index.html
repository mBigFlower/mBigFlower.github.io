异步线程

主流的有三种：

  • Thread：新线程
  • ThreadPool：线程池
  • Task：ThreadPool的优化

区别：

Thread为单线程，每次创建都会新建一个线程，这势必会造成内存增加等影响；
ThreadPool线程池，顾名思义。里面由很多线程，避免了Thread的多次新建的影响。
Task为ThreadPool的升级版，解决了ThreadPool无优先级、不能灵活取消等问题

具体的使用方法等，可以看下面两篇文章：

  • C#中 Thread，Task，Async/Await，IAsyncResult 的那些事儿！
  • 使用Task代替ThreadPool和Thread

异步线程与图片加载

首先来看下正常的图片加载：

Uri uri = new Uri("E:\\HyteraLearning\\project\\download\\imgLoadApp\\imgLoadApp\\1.jpg");//设置图片路径

BitmapImage bmp = new BitmapImage();
bmp.BeginInit();//初始化
bmp.UriSource = uri;
bmp.EndInit();//结束初始化
bmp.Freeze();

myImage.Source = bmp;

与安卓相同，UI的操作肯定要在主线程。那么最后一步肯定是要出现在主线程的，异步代码如下：

public MainWindow()
{
    ...

    Thread thread = new Thread(UpdateImage);
    thread.Start();
}

private void UpdateImage()
{
    Uri uri = new Uri("E:\\HyteraLearning\\project\\download\\imgLoadApp\\imgLoadApp\\1.jpg");//设置图片路径

    BitmapImage bmp = new BitmapImage();
    bmp.BeginInit();//初始化
    bmp.UriSource = uri;
    bmp.EndInit();//结束初始化
    bmp.Freeze();// Important !

    this.Dispatcher.Invoke(new Action(() => {
        myImage.Source = bmp;
    }));
}

可以看出，我们创建了一个Thread，并把刚刚的图片加载代码放到了Thread中，不同的是，调用了一个：

this.Dispatcher.Invoke(new Action(() => {
    myImage.Source = bmp;
}));

其目的是让图片加载的代码在主线程中执行。

同时，这里面有一句代码很重要，不然bmp不能赋值到Source中：

bmp.Freeze();// 使当前对象变为不可修改并将其 System.Windows.Freezable.IsFrozen 属性设置为 true。

意思就是告诉主线程，这个bmp我不改了，以后给你用，拿走吧。这时候主线程才敢放心的使用。
就好比安卓中，有些参数要使用final属性，才能传递到“内部”。

BeginInvoke 和 Invoke

感觉上BeginInvoke是异步，而Invoke是同步，实际上当内部执行了耗时的操作时，还是会出现假死的情况。

异步、多线程

“异步是DMA，通过回调来交互，操作复杂度较高”
“多线程是并发，操作简单，但各线程间的共享资源可能出现死锁”

“在了解了线程与异步操作各自的优缺点之后，我们可以来探讨一下线程和异步的合理用途。我认为：当需要执行I/O操作时，使用异步操作比使用线程+同步I/O操作更合适。I/O操作不仅包括了直接的文件、网络的读写，还包括数据库操作、Web Service、HttpRequest以及.Net Remoting等跨进程的调用。

而线程的适用范围则是那种需要长时间CPU运算的场合，例如耗时较长的图形处理和算法执行。但是往往由于使用线程编程的简单和符合习惯，所以很多朋友往往会使用线程来执行耗时较长的I/O操作。这样在只有少数几个并发操作的时候还无伤大雅，如果需要处理大量的并发操作时就不合适了。”
